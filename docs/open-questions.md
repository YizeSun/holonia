# 开放问题、遗漏项和决策顺序

以下内容尚未确定。它们被记录下来，但不应阻塞 NearBridge 的第一条真机链路。

## 1. NearBridge 近期必须回答

### 平台和产品形态

- 第一版是否确定为 iPhone + Mac mini？
- 使用一个跨平台 SwiftUI 工程，还是 App 与常驻 Mac Host 分开？
- Mac mini 是否需要登录后常驻，还是只在 App 前台工作？

### 发现和会话

- UDP broadcast、multicast、Bonjour 和 MultipeerConnectivity 各自承担什么职责？
- 发现和可靠会话是否使用不同技术？
- iOS 后台时需要维持到什么程度？
- 热点、企业 Wi-Fi 和禁用广播的网络如何表现？

### 身份和配对

- 节点身份与 Holon Account 身份如何关联？
- 首次配对需要双端确认、二维码还是短码？
- 设备丢失、密钥轮换和账户恢复如何处理？

## 2. Core 边界

- 哪些消息必须进入 Holonia Core，哪些属于专业网络 Profile？
- 通用 Offer、Accept、Delivery、Acceptance 是否进入 Core，还是作为第一个标准 Profile？
- Resource、Action、Policy 的扩展标识使用字符串命名空间、Schema URI 还是其他方式？
- 什么时候发布第一个可兼容的协议版本？

建议：在至少两条真实端到端链路稳定前，不承诺长期兼容。

## 3. 开放 P2P 网络

- 新 Holon 如何发现第一批邻居？
- Propagation Cluster 由谁创建、如何加入和退出？
- Relay Record 记录完整路径，还是只记录可验证跳数？
- 如何减少拓扑隐私泄露？
- 临时 Reply Route 由谁提供？
- 直连、NAT 穿透和 Relay fallback 如何组合？
- 如何处理 Sybil、垃圾广播、UDP 欺骗和放大攻击？

## 4. 私密会话和数据披露

- Session Grant 使用怎样的最小表达方式？
- Host 如何判断自然语言消息是否超出授权范围？
- 第三方 Primary Holon运行在本地进程、沙箱、扩展还是远端服务？
- 第三方实现获得什么上下文，如何证明没有保留数据？

NB-6 只回答了 spike 级最小路径：Mac App 内、编译时 allowlisted、纯文本 `HolonAdapter`，不暴露文件、网络、Host key 或动态工具。它没有解决第三方进程沙箱、Primary Holon Account 绑定、上下文保留证明或长期运行问题。

## 5. 信誉

- Publishing Reputation 与 Responding Reputation 各记录哪些事实？
- 哪些事件需要双方签名？
- 如何区分事实、评价和争议结论？
- 信誉是否按专业网络和能力类型分别计算？
- 多账户、串谋和自我交易如何处理？
- 何时限制发布、传播或回应？

暂不选择评分公式。第一要求是保存可验证原始事件并允许重新计算。

## 6. 支付与商业模式

- 代码任务按固定结果、里程碑还是其他方式定价？
- 平台向需求方、执行方还是双方收费？
- 什么时候付款，验收争议如何影响付款？
- 是否先人工结算验证需求，再实现自动支付？
- 模型 token 和算力成本由执行方内部承担，还是需要单独计量？

建议：先验证用户愿意为小型代码结果付费，不先建设通用支付系统。

## 7. 第一个代码网络

- 首个任务只支持 Swift/Apple 项目，还是接受其他语言？
- 交付采用 Patch、Commit 还是 Pull Request 为主？
- 测试由需求方环境运行还是执行方环境运行？
- 最小复现包如何扫描秘密、恶意代码和许可证风险？
- 何时支持私有仓库和隔离执行环境？

## 8. 容易遗漏但应该保留的位置

### 威胁模型

在实现开放网络前，需要明确保护对象、可信组件、攻击者能力和无法提供的安全保证。

### 审计与可观察性

从 NearBridge 第一版开始记录关键状态和错误，否则跨设备问题难以诊断，也无法为未来信誉提供事实来源。

### 协议兼容和迁移

消息应尽早携带版本，但兼容策略应在真实演化出现后制定。

### 用户撤销和数据生命周期

配对撤销、退出共享域、关闭会话、删除账户后，哪些本地数据和远端数据仍保留，需要后续明确。

### 许可证和贡献模式

在公开 GitHub 前，需要选择开源许可证、贡献规范、安全报告流程和商标使用规则。

### 产品验证

至少找出一位真实需求方和一位真实执行方，观察他们完成联系与交付，而不仅依赖开发团队自己的模拟。

## 9. 推荐的近期决策顺序

1. 确认 NearBridge 第一设备组合和 App 形态；
2. 完成发现与可靠会话技术实验；
3. 固定最小联系型消息；
4. 在真机跑通联系型需求；
5. 用一个公开仓库跑通人工代码交付；
6. 根据真实问题调整 Core 边界；
7. 再设计开放 P2P、信誉和支付。
