# 小型、可验证代码任务网络计划

更新时间：2026-07-21

状态：未来 Specialized Network 设计，尚未实现。当前 NearBridge 没有
workspace 读取、代码修改、命令执行或长期 Agent 权限；这些能力必须在
受控文件 broker、typed tool policy、逐次授权、隔离和恢复机制完成后再推进。

## 1. 收窄后的产品范围

第一个专业网络只服务：

> 已有公开代码仓库或最小复现项目中的一项小型、可验证修改。

首选任务：

- 修复一个明确 Bug；
- 添加一个边界清楚的小功能；
- 补充或修复测试；
- 修复构建问题；
- 完成一次范围明确的代码审查。

暂不接受：

- 从零开发完整商业 App；
- 无边界的长期外包；
- 企业生产环境操作；
- 含大量秘密的私有仓库；
- 无法描述完成条件的开放式咨询。

## 2. 必须先包含联系型需求

在代码交付之前，先完成一个不含正式委托的联系型流程：

```text
用户：寻找能够修复 SwiftUI 状态问题的开发能力
→ Request 发布
→ 候选 Holon 回应
→ 用户确认其中一个回应是有效联系
→ Request Completed
```

这一阶段验证：

- 非技术用户是否能形成一个可传播摘要；
- 需求类型是否足以帮助初步筛选；
- 回应是否相关；
- 从公开传播进入私密会话是否自然；
- 联系型需求是否有独立价值。

## 3. 第一条交付型任务

推荐标准用例：

> 修复一个公开仓库或最小复现项目中的明确 Bug，并以 Patch、Commit 或
> Pull Request 交付。

### 输入

- 仓库 URL 或最小复现包；
- 问题说明；
- 复现步骤；
- 期望行为；
- 可选测试命令。

### 输出

- Patch、Commit 或 Pull Request；
- 修改说明；
- 测试或构建结果；
- 已知限制。

### 验收

- 原问题不再出现；
- 指定测试或构建通过；
- 没有明显破坏既有行为；
- 需求方或指定 Reviewer 确认。

## 4. 用户流程

```text
选择仓库或最小复现
→ Primary Holon 帮助澄清问题
→ 生成脱敏 Capability Request
→ Host 和用户批准传播
→ 收到 Capability Response
→ 建立私密会话
→ 需求方发起简单 Engagement Offer
→ 执行方接受
→ 执行与交付
→ 按 Acceptance Policy 验收
→ Completed
```

## 5. 第一阶段的 Engagement

只要求：

```text
Deliverable Specification
Acceptance Policy
Reviewer
Signed Version
```

价格和付款暂时可以人工记录，不在协议内自动结算。

第一版不实现复杂分包界面。Core 设计不阻止后续分包，但代码网络原型
默认由一个执行 Holon 完成交付。

## 6. 代码访问策略

第一版只支持：

1. 公开仓库；
2. 用户主动提供的最小复现项目。

暂不支持：

- 对完整私有仓库的持续权限；
- 远程控制用户开发机；
- 企业密钥和生产凭证；
- 无隔离的远程命令执行。

后续候选包括临时分支权限、受限仓库令牌和需求方设备上的隔离工作区。

## 7. 里程碑

### CN-0：单机流程模拟

不联网，使用两个测试 Holon 完成 Request、Response、Contact Accepted 的状态流。

### CN-1：NearBridge 联系闭环

在 iPhone 与 Mac mini 之间完成真实联系型需求。

### CN-2：私密会话和简单 Offer/Accept

验证 Session Grant、用户接管和正式委托成立。

### CN-3：人工代码交付

执行者可以是人类高手使用 Codex；平台只记录交付和验收，不自动执行代码。

### CN-4：受控 Agent 执行

允许已授权 Agent 对公开仓库或最小复现进行处理，并返回 Patch。

### CN-5：小范围真实付费验证

通过人工结算验证用户是否愿意为结果付款，再决定支付协议。

## 8. 成功指标

第一阶段重点观察：

- 从发布到第一个有效回应的时间；
- 有效回应比例；
- 从回应到正式委托的比例；
- 任务完成率；
- 验收通过率；
- 用户为形成需求投入的时间；
- 用户是否愿意再次发布和付费；
- 执行方是否认为任务边界足够清楚。

不以注册用户数或广播总量作为早期核心指标。
